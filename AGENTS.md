# AGENTS.md — guidance for AI assistants

This file orients AI coding assistants working on **open-balena-extension**. Read it before making changes. (`CLAUDE.md` points here.)

## What this project is
Production extensions for self-hosted [openBalena](https://github.com/balena-io/open-balena): a **remote builder** (so `balena push` works), an **imagemaker** (web UI + scripts to build flashable device images), a **custom haproxy** ingress, and integration with the community **admin dashboard**. Core openBalena is unmodified upstream; everything here sits *around* it.

## The one mental model you must hold: internal vs public TLD
openBalena issues device endpoints under a single `DNS_TLD`. This project supports running an **internal** `DNS_TLD` (e.g. `example.local`) while reaching the instance **publicly** via a reverse proxy at `PUBLIC_TLD` (e.g. `ob.example.com`).
- The API advertises `api/cloudlink/registry2/delta.${DNS_TLD}` to devices.
- Remote devices can't resolve `*.${DNS_TLD}` → their `config.json` endpoints must be rewritten to `*.${PUBLIC_TLD}`.
- The imagemaker does this for its **`__internet`** image variant (`sed s/${DNS_TLD}/${PUBLIC_TLD}/g`); the **`__lan`** variant keeps internal names for on-LAN devices.
- **Symptom of getting it wrong:** a device shows `api_heartbeat_state: online` (API reachable) but `is_connected_to_vpn: false` forever (VPN endpoint still internal). See [docs/dns-tld-split.md](docs/dns-tld-split.md).

## Hard invariants — do not break these
1. **Builder token must match the API.** `TOKEN_AUTH_BUILDER_TOKEN` on the builder/delta MUST equal the core API's value (it's mirrored to `REGISTRY2_TOKEN`). It is generated by the API into the node process env; read it from `/proc/<node-pid>/environ`, not `docker exec printenv` (that shows only image-time env).
2. **Builder needs a plaintext Docker daemon.** The community builder CLI speaks only plaintext `tcp://host:2375`. Use the dedicated `builder-dind` (`DOCKER_TLS_CERTDIR=""`), not the TLS test dind.
3. **The bundled builder CLI expires.** `obcommunity/open-balena-builder` ships an old `balena-cli` that hard-expires (~365 days after the next major). The patched image swaps in a current standalone CLI and wraps it to export `HOME` + `NODE_TLS_REJECT_UNAUTHORIZED=0` (the builder spawns the CLI with a stripped env, so these MUST be in the wrapper, not the container env).
4. **One `balena push` at a time.** Concurrent builds race on release/service creation and fail with `"application" and "service_name" must be unique`.
5. **haproxy is the single ingress.** Validate any `haproxy.cfg` change with `haproxy -c` inside the running container (it has the certs + env) BEFORE reloading; keep a timestamped backup. The builder route needs `timeout server 3600s` (builds exceed the default 63s).
6. **Never commit secrets.** Templatize to `${ENV}`: `TOKEN_AUTH_BUILDER_TOKEN`, `IMAGEMAKER_TOKEN`, `OPEN_BALENA_JWT_SECRET`, `OPEN_BALENA_S3_*`, `PGRST_JWT_SECRET`, any `*.key`/CA/`certs/`. The `.gitignore` blocks these — keep it that way.
7. **The imagemaker is sensitive.** Its images embed fleet-provisioning credentials, so it must stay authenticated (`IMAGEMAKER_TOKEN`) and/or loopback-only. Keep its `sudo` allowlist scoped — no blanket `cp`/`docker` (each is host-root); the DB read goes through `/usr/local/bin/ob-fleets`.

## Device access reality (important when "fixing a device")
- `balena device ssh` and the dashboard terminal both ride the **VPN**. If a device's VPN is down, neither works.
- Host SSH (`:22222`) only trusts keys baked into `config.json` `os.sshKeys` at **provision time**; this OS does **not** sync newly-registered account keys to a running device (even after reboot). So you cannot mint a key a misconfigured device will accept — fix via console or **reflash** with a correct `__internet` image (which also bakes in registered account keys).

## Repo layout
- `components/` — the actual source (imagemaker, builder, haproxy, admin). Reference implementations; paths/domains parameterized where practical.
- `ansible/` — playbooks/roles to deploy core + extensions.
- `docs/` — architecture, the DNS split, setup, runbook, CLI onboarding.

## When editing
- Keep `components/` source and the `ansible/` roles in sync (a change to a script should be reflected in the role that deploys it).
- Prefer parameterizing new hardcoded hostnames/paths via the `.env` / Ansible vars (`dns_tld`, `public_tld`, `install_root`, `service_user`).
- Update the relevant `docs/` and this file when you change an invariant.
